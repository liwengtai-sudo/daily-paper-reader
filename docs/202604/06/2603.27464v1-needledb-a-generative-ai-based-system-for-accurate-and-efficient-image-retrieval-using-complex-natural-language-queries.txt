ABSTRACT 

We demonstrate NeedleDB , an open-source, deployment-ready database system for answering complex natural language queries over image data. Unlike existing approaches that rely on contrastive-learning embeddings (e.g., CLIP), which degrade on compositional or nuanced queries, NeedleDB leverages generative AI to syn-thesize guide images that represent the query in the visual do-main, transforming the text-to-image retrieval problem into a more tractable image-to-image search. The system aggregates nearest-neighbor results across multiple vision embedders using a weighted rank-fusion strategy grounded in a Monte Carlo estimator with provable error bounds. NeedleDB ships with a full-featured command-line interface ( needlectl ), a browser-based Web UI, and a modular microservice architecture backed by PostgreSQL and Milvus. On challenging benchmarks, it improves Mean Average Precision by up to 93% over the strongest baseline while maintaining sub-second query latency. In our demonstration, attendees interact with Nee-dleDB through three hands-on scenarios that showcase its retrieval capabilities, data ingestion workflow, and pipeline configurability.

1 INTRODUCTION 

Answering natural language queries over image collections is a fun-damental challenge in multi-modal data management. State-of-the-art approaches use contrastive-learning models such as CLIP [ 12 ]and ALIGN [ 7 ] to project both text and images into a shared embed-ding space and then apply nearest-neighbor search. While effective for simple object-detection queries (e.g., “ traffic light ”), these meth-ods break down for complex, compositional queries that involve spatial relationships, attribute descriptions, or abstract concepts [ 4]. Consider the queries of increasing complexity shown in Fig-ure 1: while CLIP retrieves relevant images for a simple query such as “traffic light” , it fails entirely on the moderate query “a metro wagon” and returns mostly irrelevant results for “a person walk-ing with a cane” . In contrast, NeedleDB retrieves relevant images for all three queries from the BDD100K dataset [ 19 ], using only two guide images generated by RealVisXL and two vision embed-ders (EVA [ 5 ], RegNet [ 18 ]). 2.2 Inference Pipeline 

At query time, NeedleDB processes each query through an opti-mized two-stage pipeline: (1) Guide Image Generation and Multi-Embedder Search. The system invokes the Generator Hub service to produce 𝑚 syn-thetic guide images using one or more text-to-image foundation models (e.g., SD-Turbo [ 14 ], DALL-E 3 [ 3 ]). By using locally hosted latent-consistency models such as SD-Turbo with 1–4 inference steps at 512 ×512 resolution, generation completes in milliseconds rather than seconds. Each guide image is then embedded by 𝑙 vision models selected at installation time, and a 𝑘 -NN search is performed per (guide image, embedder) pair. (2) Weighted Rank Aggregation. Results from all (𝑚 ×𝑙 ) searches are combined using weighted reciprocal-rank fusion, where each embedder’s weight reflects its topic-specific reliability. An anomaly detection module based on Local Outlier Factor filters low-quality guide images before aggregation. 

2.3 System Features 

One-Liner Installation. NeedleDB can be installed on Linux or macOS with a single command that detects the host OS and GPU (NVIDIA CUDA or Apple MPS), downloads all dependencies, spins up Docker infrastructure, and installs the needlectl binary: 

curl -fsSL https://raw.githubusercontent.com/ UIC-InDeXLab/Needle/main/scripts/ install-oneliner.sh | bash -s fast 

Command-Line Interface ( needlectl ). needlectl is a single-file binary built with PyInstaller, distributed via GitHub Releases for Linux and macOS, so users need no Python environment to operate the system. It provides five command groups spanning the full lifecycle (Figure 4): 

Service management. [needlectl service start | stop | restart | status] orchestrates all infrastructure containers and Python mi-croservices. [needlectl service log] streams logs from the backend, generator hub, or Docker infrastructure. 

[needlectl service update] performs in-place upgrades of in-dividual components (the CLI binary itself, the backend, or the Web UI) by downloading the latest GitHub release artifacts, en-abling zero-downtime updates without reinstallation. 

Directory management. [needlectl directory add <path>] reg-isters an image directory and optionally shows a live indexing progress bar. [needlectl directory list | describe | modify | remove] provide CRUD operations over directories. 

Query execution. needlectl query run "a dog on a surfboard" -n 10 issues a search query and returns ranked results with a pre-view URL (Figure 4a); verbose output shows per-embedder rank-ings. 

Generator and UI management. [needlectl generator list | config] 

display and configure the text-to-image engines. [needlectl ui start | stop | build] manage the React-based Web UI. 

Terminal UI (TUI). Several needlectl subcommands ( [service config] , [directory modify] , [generator config] ) launch in-teractive Textual-based terminal editors (Figure 4c, 4d) where users toggle boolean flags, edit environment variables, reorder generators by priority, and save changes—all without leaving the terminal. 

Versioning and Upgradability. NeedleDB uses Git-tag-based semantic versioning: each push to main triggers a CI/CD pipeline that computes the version from the latest [needlectl/v*] tag plus the number of commits since, then builds and publishes platform-specific binaries. Running [needlectl –version] reports the version of each component (CLI, backend, UI). The [needlectl service update] command selectively upgrades any component by fetching the corresponding release artifact, making it straight-forward to roll out patches or new embedder models without a full reinstallation. 

Parallel Indexing. The backend’s IndexQueueManager uses a prior-ity queue backed by a configurable thread pool (default: 4 workers). Directories are processed in parallel, and within each directory, images are batched (default: 50 images per batch) so that each em-bedder performs a single GPU forward pass per batch. A FileWatch-erService based on the watchdog library monitors registered direc-tories in real time: newly added images are automatically queued for indexing, deleted images are removed from both PostgreSQL and Milvus, and modified images are re-embedded. A periodic Con-sistencyChecker reconciles the database and filesystem states to handle changes that occur while the system is offline. 

Easy Integration of New Embedders. Adding a new vision em-bedder requires only appending an entry to the embedders.json

Web UI. A browser-based React interface (Figure 3) offers: (a) a 

search page (Figure 3a) where users type natural language queries and configure generation parameters (number of guide images, en-gines, image resolution), with results displayed alongside the gener-ated guide images; (b) a directory management page (Figure 3c) with live indexing progress bars, enable/disable toggles, auto-refresh, and per-directory image counts; (c) a generator configuration page

Modular Image Generation. The Generator Hub microservice supports six text-to-image engines out of the box—DALL-E [ 3], Replicate, RealVisXL, Imagen3-fast, local SD-Turbo [ 14 ], and a generic Local HTTP endpoint—with a priority-queue-based fall-back mechanism: if the highest-priority engine fails, the request is transparently rerouted to the next available engine. 

Configuration Modes. The user selects one of three pre-tuned profiles at installation time, which determines the set of vision embedders, generation parameters, and index settings for the entire deployment: 

• Fast — 2 embedders (EVA, RegNet), 1 guide image at 512 ×512 ;optimized for low-latency retrieval. 

• Balanced — 4 embedders, 2 guide images; a middle ground between speed and accuracy. 

• Accurate — 6 embedders with learned weights, 2 guide images; highest retrieval quality. Since the choice of embedders determines which Milvus collections are created and which models are loaded, the mode is fixed at installation time to ensure a consistent index. The Fast configuration achieves a total inference time of 0.203 s per query on a 100K-image dataset, competitive with CLIP’s 0.184 s while delivering substantially higher accuracy [4]. 

3 DEMONSTRATION SCENARIOS 

We design three interactive scenarios that highlight NeedleDB ’s capabilities. A laptop pre-loaded with the LVIS dataset [ 6] ( ∼120K images, 1000+ categories) will serve as the demonstration platform. 

3.1 Scenario 1: Simple vs. Complex Queries 

Goal: Illustrate the core value proposition of NeedleDB .An attendee first issues a simple query such as “ a red car ” and observes that both CLIP and NeedleDB return satisfactory results. The attendee then composes a complex query—for example, “ a cat wearing a tiny cowboy hat sitting on a stack of books ”—and sees CLIP fail to retrieve any relevant image while NeedleDB returns accurate results. The interface displays the generated guide images alongside the retrieved results (Figure 3a), making the generation-based re-trieval mechanism visually transparent. Attendees are encouraged to compose their own queries of varying complexity and observe how NeedleDB ’s locally optimized SD-Turbo pipeline delivers sub-second responses even for challenging compositional queries.

Goal: Demonstrate NeedleDB ’s data ingestion workflow. The attendee uses the Web UI to register a new image directory (either a pre-loaded collection or one brought on a USB drive). The directory management page (Figure 3c) displays a live progress bar as NeedleDB scans, embeds, and indexes the images. 3.3 Scenario 3: Under the Hood 

Goal: Enable technically inclined attendees to explore NeedleDB ’s configurability. Using the CLI, the attendee varies the number of guide images (𝑚 ∈ { 1, 3, 9}) and the image generation quality ( SMALL , MEDIUM ,

LARGE ). The results panel shows individual per-embedder rankings alongside the aggregated output, making the rank-fusion process transparent. We also compare results from different foundation models (local SD-Turbo vs. API-based DALL-E 3; Figure 3d) and demonstrate how the ensemble of generators mitigates individual model biases—for instance, the “bag-of-words” bias that ignores compositional ordering. 

4 EXPERIMENTAL HIGHLIGHTS 

We summarize key results from the full experimental evaluation presented in [4]. 

Object Detection. Table 1 reports MAP on four benchmarks. Nee-dleDB outperforms all eight baselines, with particularly large gains on hard queries (those with CLIP AP < 0.5): +73% on Caltech256 and +93% on LVIS relative to the second-best baseline. 

Complex Natural Language Queries. On the COLA [ 13 ], Winoground [ 15 ], SentiCap [ 10 ], and NoCaps [ 1] benchmarks, NeedleDB achieves the highest Pairing Accuracy (0.631 on COLA, 0.593 on Winoground) and MRR (0.642 on SentiCap, 0.745 on NoCaps), outperforming all contrastive-learning baselines [4]. 

Latency. The optimized Fast pipeline (SD-Turbo + EVA + RegNet) processes a query in 0.203 s end-to-end on the LVIS dataset (100K images), of which 0.136 s is guide image generation and 0.053 s is multi-embedder search and retrieval. 

Contribution Decomposition. An ablation study isolates the sources of improvement: switching from CLIP-based text-to-image to NeedleDB ’s generative framework (using CLIP as the sole em-bedder) yields a 35% relative MAP gain on LVIS; replacing CLIP with a stronger image-only embedder (EVA) adds a further 20%; and the full ensemble provides additional incremental gains [4].

5 CONCLUSION 

We demonstrated NeedleDB , an open-source database system that uses generative AI to answer complex natural language queries over image datasets. By transforming text-to-image retrieval into image-to-image search and aggregating across multiple embedders, 

NeedleDB substantially outperforms existing contrastive-learning approaches while maintaining practical query latencies. Future work includes extending the framework to audio/video and improv-ing foundation model alignment for compositional queries.
